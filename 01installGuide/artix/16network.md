##### iwd
 Only root and members of the network or the wheel user group are allowed
 to interact with iwd. In order to use iwctl or other front-ends,
 you need to add your user to one of those groups.

```bash
pacman -S iwd iwd-openrc --needed
rc-status
rc-service iwd start
rc-update add iwd default

sudo mkdir -p /etc/iwd
sudo nvim /etc/iwd/main.conf

[General]
EnableNetworkConfiguration=true

There is also ability to set route metric with RoutePriorityOffset:

sudo nvim /etc/iwd/main.conf

[Network]
RoutePriorityOffset=300

!!! IMPORTANT !!!
echo "nameserver 1.1.1.1" > /etc/resolv.conf
```

```
/etc/iwd/main.conf
[Network]
RoutePriorityOffset=300

[General]
EnableNetworkConfiguration=true
```

Since you can reach 1.1.1.1, the problem is specifically DNS resolution.

First check what DNS server was configured:

cat /etc/resolv.conf

Typical working output:

nameserver 1.1.1.1

or

nameserver 192.168.1.1

If the file is empty, missing, or contains invalid entries, DNS won't work.

Also check what iwd thinks the DNS configuration is:

iwctl station wlan0 show

Look for lines such as:

Connected network
IPv4 address
IPv4 gateway
IPv4 DNS

The important one is IPv4 DNS.

Because you are using:

[General]
EnableNetworkConfiguration=true

iwd should be creating /etc/resolv.conf itself.

Verify that:

ls -l /etc/resolv.conf

If it's a dangling symlink or points somewhere that doesn't exist, DNS will fail.

For example:

cat /etc/resolv.conf

might show:

# Generated by iwd
nameserver 192.168.1.1

which should work.

A quick test:

echo "nameserver 1.1.1.1" > /etc/resolv.conf
ping artixlinux.org -c2

If that immediately starts working, you've confirmed the issue is how DNS information is being written into resolv.conf.

Please show the output of:

cat /etc/resolv.conf

ls -l /etc/resolv.conf

iwctl station wlan0 show

and we can pinpoint exactly where the DNS configuration is failing.
